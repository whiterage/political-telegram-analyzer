# Telegram Emotion Analyzer

Go-сервис для анализа Telegram-постов: собирает публикации, считает метрики вовлеченности, определяет эмоциональный фрейм текста, анализирует реакцию аудитории по эмодзи и выгружает результаты в CSV для дальнейшего исследования.

Проект сделан как прикладной аналитический инструмент на стыке backend-разработки, data processing и медиа-аналитики. Его можно уверенно показывать в резюме как пример системы, где из сырых данных строится воспроизводимый пайплайн с доменной моделью, валидацией, конфигурацией, агрегированием и экспортом результатов.

## Зачем этот проект

Задача проекта: превратить разрозненные Telegram-посты в структурированный аналитический датасет, по которому можно сравнивать каналы, форматы публикаций и эмоциональные паттерны контента.

Практическая ценность:

- сравнение постов не по абсолютным реакциям, а по normalized engagement через ERR;
- выделение эмоциональных фреймов текста и эмоционального отклика аудитории;
- агрегация результатов по каналам, категориям и типам акторов;
- подготовка данных для исследований, отчетов, BI и последующей ML/LLM-аналитики.

## Что реализовано

На текущем этапе сервис уже решает полный end-to-end сценарий для JSON-источника:

- загружает посты из JSON;
- валидирует входные данные;
- фильтрует посты по временному диапазону;
- отсекает шум по порогам `min_views` и `min_total_reactions`;
- считает `total_reactions` и `ERR`;
- определяет формат поста по длине текста;
- классифицирует эмоциональный фрейм текста через rule-based классификатор;
- определяет доминирующую эмоцию реакции аудитории по эмодзи;
- выбирает top-N постов внутри каждого канала;
- строит агрегированные сводки по эмоциям, каналам и группам каналов;
- экспортирует результаты в несколько CSV-артефактов.

Дополнительно уже подготовлена архитектурная база под второй источник данных:

- Telegram/MTProto source через `gotd/td`;
- конфигурация каналов, категорий и actor type;
- env overrides для `TELEGRAM_API_ID`, `TELEGRAM_API_HASH`, `TELEGRAM_SESSION_FILE`.

Сейчас Telegram source уже умеет авторизоваться и читать историю канала, но маппинг сообщений в `domain.Post` еще не завершен. Это честно отражено в коде и не маскируется в документации.

## Архитектура

Проект разбит на независимые модули, что делает его удобным для расширения и тестирования:

- `cmd/analyzer` - точка входа приложения;
- `internal/app` - инициализация приложения и загрузка конфига;
- `internal/pipeline` - orchestration всего аналитического пайплайна;
- `internal/source` - абстракция источников данных (`json`, `telegram`);
- `internal/domain` - доменные модели постов и результатов анализа;
- `internal/filter` - фильтрация по датам и порогам качества данных;
- `internal/validation` - проверка консистентности входных сущностей;
- `internal/analysis` - расчет метрик, формата и ranking логики;
- `internal/emotion` - интерфейсы и реализации классификации эмоций;
- `internal/reactionanalysis` - анализ эмоций аудитории по реакциям;
- `internal/summary` - агрегированные сводки;
- `internal/export` - экспорт результатов в CSV.

Схема пайплайна:

```text
source -> filtering -> validation -> analysis -> ranking -> summaries -> CSV export
```

## Ключевые аналитические сущности

### ERR

Используется метрика `ERR` (Engagement Rate by Reach):

```text
ERR = (reactions + comments + forwards) / views * 100
```

Она позволяет сравнивать вовлеченность постов корректнее, чем сырые просмотры или реакции.

### Эмоциональный анализ

Сейчас в проекте реализован rule-based классификатор, который определяет текстовые фреймы, например:

- `anger_accusation`
- `fear_threat`
- `pride_victory`
- `sadness_empathy`
- `ridicule_discreditation`
- `neutral`

Отдельно анализируются реакции аудитории по эмодзи с выделением доминирующей реакции и ее эмоциональной интерпретации, например `approval`, `anger`, `sadness`, `ridicule`, `shock`.

### Агрегации

На выходе формируются не только топовые посты, но и сводки:

- по эмоциональным фреймам текста;
- по dominant reaction emotion;
- по отдельным каналам;
- по группам каналов: `channel_category` и `channel_actor_type`.

## Выходные артефакты

После запуска сервис формирует несколько CSV-файлов:

- `out_posts.csv` - детальная выгрузка top-posts с метриками, эмоциями и метаданными;
- `out_summary.csv` - агрегированная сводка по эмоциям и reaction patterns;
- `out_channel_summary.csv` - средние метрики по каналам;
- `out_channel_group_summary.csv` - сравнение групп каналов по категориям и типам акторов.

Это делает проект полезным не только как backend-сервис, но и как data-preparation слой для аналитики и визуализации.

Из корня репозитория:

```bash
go run ./cmd/analyzer
```

По умолчанию используется конфигурация из [config.yaml](/Users/zahar/it/sofiasoft/config.yaml) и тестовый JSON-датасет [posts.json](/Users/zahar/it/sofiasoft/posts.json).

## Конфигурация

Основные параметры задаются в `config.yaml`:

- источник данных: `json` или `telegram`;
- входной и выходные файлы;
- временное окно анализа;
- пороги `min_views` и `min_total_reactions`;
- `top_limit` для отбора лучших постов внутри каждого канала;
- список каналов с доменными атрибутами;
- Telegram credentials и session settings.

Для Telegram credentials поддержаны env overrides:

```bash
export TELEGRAM_API_ID=...
export TELEGRAM_API_HASH=...
export TELEGRAM_SESSION_FILE=telegram.session
```
